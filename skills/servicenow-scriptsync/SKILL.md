# sn-scriptsync Integration Guide

## When to Use sn-scriptsync vs MCP Tools

**Rule of thumb:** If you are reading or writing *data*, use MCP tools. If you are syncing *script files*, use sn-scriptsync.

| Task | Use |
|------|-----|
| Read table schema | MCP: `table_describe` |
| Query records | MCP: `build_query` + `table_query` |
| Create/update data records | MCP: `record_create` / `record_update` |
| Edit a Business Rule script | sn-scriptsync: edit local file + `sync_now` |
| Create a new Script Include | sn-scriptsync: `create_artifact` |
| Read an existing script body | MCP: `meta_get_artifact` (to read) or sn-scriptsync (to get the file locally) |

## File Structure

sn-scriptsync organizes files as:

```
<instance>/<scope>/<table>/<artifact_name>.script.js
```

Example:
```
dev12345/global/sys_script_include/IncidentUtils.script.js
dev12345/global/sys_script/Set Priority on Insert.script.js
dev12345/x_myapp/sys_script_include/MyAppUtil.script.js
```

### File Types

Only create files for CODE fields:
- `.script.js` -- Script body (Business Rules, Script Includes, etc.)
- `.template.html` -- HTML templates (UI Pages, email templates)
- `.css.scss` -- Stylesheets (Service Portal widgets, UI Pages)

### Managed Files -- Do Not Touch

- `_map.json` files are auto-generated by sn-scriptsync. **Never edit them manually.** They contain mappings between local files and instance records.

## Session Workflow

### Starting a Session

Always follow this sequence before working on scripts:

1. **`check_connection`** -- Verify sn-scriptsync can reach the instance. Always run first.
2. **`clear_last_error`** -- Clear any stale error state from a previous session.
3. **`switch_context`** -- Set the correct application scope if needed (e.g., switching from Global to a scoped app).
4. Begin working on scripts.

### During a Session

- Edit scripts locally in your editor. sn-scriptsync watches for file changes.
- Use `create_artifact` to create new artifacts (creates both the instance record and the local file).
- Use `open_in_browser` to view the artifact in the ServiceNow UI when needed.
- Use `take_screenshot` to capture the current state of the instance UI.

### Ending a Session

Always follow this sequence before reporting work as complete:

1. **`sync_now`** -- Push all local changes to the instance. Always run before finishing.
2. **`get_last_error`** -- Check for any sync errors.
3. Report results to the user, including any errors.

## Key Commands Reference

| Command | Purpose | When to Use |
|---------|---------|-------------|
| `check_connection` | Verify instance connectivity | Always first |
| `clear_last_error` | Clear stale errors | After check_connection |
| `switch_context` | Change application scope | When working in a scoped app |
| `sync_now` | Push local changes to instance | Always before finishing |
| `get_last_error` | Check for sync errors | After sync_now |
| `create_artifact` | Create new script artifact | Creating new BR, SI, etc. |
| `open_in_browser` | Open artifact in ServiceNow UI | Debugging, manual verification |
| `take_screenshot` | Capture instance UI state | Documentation, verification |

## Rules

1. **Never** use browser automation tools (Playwright, etc.) for ServiceNow platform interaction -- use sn-scriptsync commands instead.
2. **Never** edit `_map.json` files -- they are auto-generated and managed by sn-scriptsync.
3. **Always** run `sync_now` before reporting work as complete.
4. **Always** run `check_connection` before starting any sn-scriptsync operation.
5. **Always** check `get_last_error` after `sync_now` to confirm no sync failures.
6. Non-code fields (short_description, conditions, active flag, etc.) are managed via MCP tools or the platform UI, not through sn-scriptsync file edits.

## Error Recovery

If a sn-scriptsync command fails:

1. Read the error message carefully
2. Re-read the command format in the AGENTS-API-REFERENCE.md for the correct request structure
3. Fix the request payload and retry (up to 3 attempts)
4. If still failing after 3 retries, STOP and report the error to the user with the full error details
5. **NEVER fall back to MCP `record_create` for script artifacts** -- a broken sn-scriptsync request is fixable; an orphaned MCP record breaks workspace sync

### Common Mistakes

| Mistake | Symptom | Fix |
|---------|---------|-----|
| Using `"args"` instead of `"params"` as top-level key | `"Missing required param: table"` even though table is present | Change `"args": { ... }` to `"params": { ... }` |
| Missing `"name"` inside `fields` object | Record created without a name, or creation fails | Add `"name": "ArtifactName"` to `fields` -- required for ALL tables |
| Boolean values as `true` instead of `"true"` | Field not set correctly on the instance | Use string values: `"active": "true"`, not `"active": true` |
| Missing `"scope"` parameter | Artifact created in wrong scope or creation fails | Always include `"scope": "global"` or `"scope": "x_myapp_scope"` |
| Omitting `"id"` field in request | Request may not be trackable in response queue | Always include a unique `"id"` string |

## Lessons Learned (Real Failures)

These are documented from real agent sessions where things went wrong.

### Failure 1: `args` vs `params` key mismatch

**What happened:** Agent sent `create_artifact` with `"args": { "table": "sys_script_include", ... }` instead of `"params": { ... }`. sn-scriptsync returned `"Missing required param: table"` because it only reads from the `params` key.

**Root cause:** Agent used an incorrect request format from memory instead of consulting the API reference.

**Lesson:** Always use `"params"` as the top-level key. If you get `"Missing required param"` for a param you DID include, check the top-level key first.

### Failure 2: Fabricating reasons to bypass sn-scriptsync

**What happened:** After `create_artifact` failed (due to the `args`/`params` bug), the agent claimed "the VS Code extension might not be active" and fell back to MCP `record_create` -- despite having just received `{"ready": true, "browserConnected": true}` from `check_connection`.

**Root cause:** Agent rationalized a workaround instead of diagnosing the actual error.

**Lesson:** A successful `check_connection` means sn-scriptsync IS available. Never fabricate connectivity excuses. Fix the request, don't bypass the tool.

### Failure 3: MCP `record_create` produces orphaned records

**What happened:** The agent used MCP `record_create` to create a Script Include. The record appeared on the instance, but sn-scriptsync had no local file and no `_map.json` entry. The workspace was out of sync -- `sync_now` had nothing to push, and the agent couldn't edit the script locally.

**Root cause:** MCP write tools bypass the sn-scriptsync file tracking system entirely.

**Lesson:** Script artifacts MUST go through sn-scriptsync so that `_map.json` is updated and local files are created. MCP write tools are for data records only.
